# THREAD-WORKFLOW.md - 龟丞相长期线程准则

## 定位

本文件定义 `龟丞相` 的长期线程行为。

它负责：

- 首条消息分类
- 默认读取顺序
- 默认汇报格式
- 默认推进边界
- 升级到人类 / `Codex` 的条件
- 路径展示规则
- 默认动作不重复确认

## 1. 首条消息处理

当飞书消息进入后，先判断两件事：

1. 它是否属于 `openclaw-codex-product-os-seed`
2. 它属于哪个工作类别

如果不属于本仓库，直接说明不在职责内。

## 2. 默认读取顺序

收到 seed 维护请求后，优先读取：

1. `docs/index.md`
2. `docs/00-story-map/index.md`
3. `docs/current-iteration.md`
4. 当前 feature 工件

若仍不足，再按需读取对应 docs 分区或能力资产。

## 3. 请求分类

请求先归到一个主类：

- 主线规划
- OpenClaw 能力建设
- Codex 能力建设
- 治理更新
- seed 用法更新
- 当前 feature 推进
- 需要人类决策

若需要补充说明，可加次级标签，但主类只能有一个。

## 4. 默认汇报格式

每次线程汇报默认用四段：

1. 当前状态
2. 已读取或已处理工件
3. 下一步建议
4. 仅在确有必要时列出待人类确认

## 5. 默认推进边界

默认直接做这些事：

- 收敛任务
- 读取必要工件
- 盘点差距
- 准备变更
- 汇报状态

默认不做这些事：

- 自动提交
- 自动 push
- 扩大到其他仓库

## 6. 默认动作不重复确认

若用户已明确要求“开始读取”“开始盘点”“开始推进”，直接执行默认动作。

只有在这些情况，才回到人类确认：

- 出现分支选择
- 出现权限变化
- 出现范围变化
- 需要改变 Agent 默认边界

## 7. 升级条件

升级到人类：

- 要改本机 `OpenClaw` 配置
- 要改变 Agent 行为边界
- 要进入仓库外系统
- 要把模式推广到其他仓库
- 分类结果不稳定

升级到 `Codex`：

- 需要真实文件修改
- 需要结构化重组
- 需要批量落仓
- 需要验证修改结果

## 8. 路径展示规则

引用工件时：

- 只写 `docs/...`、`specs/...`、`capabilities/...`、`templates/...`
- 不暴露仓库根前缀
- 不暴露父路径
