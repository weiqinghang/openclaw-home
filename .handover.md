## 当前关键判断

### 1. 真正重的不是人格文件，而是 skills snapshot

当前启动成本的大头，是运行时注入的技能快照。  
人格文件只是次级成本。

### 2. 4 个 Agent 的首轮约束已基本补齐

- `taibai`：最完整
- `wukong`：已补 `BOOTSTRAP.md`，长期记忆已清噪
- `guanyin`：已补 `BOOTSTRAP.md`，工具边界已补
- `guichengxiang`：仍较完整

### 3. Agent 间协作开始进入真实链路阶段

当前至少已有一条真实可用链路：

- `guichengxiang -> ACP -> Codex`

补充：

- 项目级 ACP 直连 Codex 已可用
- OpenClaw 内部 ACP runtime 已通过本地补丁打通
- 但该能力仍受上游修复进度影响，不能视为“已无运维负担”

### 4. skill 目录必须自包含，不要把业务实现散在仓库根

`SKILL.md`、业务脚本、reference、asset 应优先同目录收口。  
仓库级 `scripts/` 只留公共基础设施。

### 5. 会驱动 Agent 的文档，必须和人类说明文档分开

否则长期维护后，Agent 输入边界会越来越脏。

### 6. 多 Agent 安全底线不能只散落在各自人格文件

仅靠单个 Agent 的提示词，拦不住运行态自毁命令。  
需要“全局公约 + 技术保险丝”双层约束。

### 7. Feishu 多账号模式下，群 allowlist 必须写到账号级

这是 2026-03-14 新验证出的关键坑。

错误直觉：

- 只写顶层 `channels.feishu.groupAllowFrom`
- 只写顶层 `channels.feishu.groups.<groupId>`

在多账号 Feishu 模式下，这样可能出现：

- `config hot reload applied`
- 但运行时仍报：
  - `group oc_xxx not in groupAllowFrom (groupPolicy=allowlist)`

本次确认的正确写法：

- `channels.feishu.accounts.<accountId>.groupPolicy`
- `channels.feishu.accounts.<accountId>.groupAllowFrom`
- `channels.feishu.accounts.<accountId>.groups.<groupId>`

其中：

- `groupPolicy = "allowlist"`
- `groupAllowFrom += groupId`
- `groups.<groupId>.requireMention = false`
- `groups.<groupId>.allowFrom = [ownerUserId]`
