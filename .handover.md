### 11. Feishu 多账号模式下，群 allowlist 必须写到账号级

这是 2026-03-14 新验证出的关键坑。

错误直觉：

- 只写顶层 `channels.feishu.groupAllowFrom`
- 只写顶层 `channels.feishu.groups.<groupId>`

在多账号 Feishu 模式下，这样可能出现：

- `config hot reload applied`
- 但运行时仍报：
  - `group oc_xxx not in groupAllowFrom (groupPolicy=allowlist)`

本次确认的正确写法：

- `channels.feishu.accounts.<accountId>.groupPolicy`
- `channels.feishu.accounts.<accountId>.groupAllowFrom`
- `channels.feishu.accounts.<accountId>.groups.<groupId>`

其中：

- `groupPolicy = "allowlist"`
- `groupAllowFrom += groupId`
- `groups.<groupId>.requireMention = false`
- `groups.<groupId>.allowFrom = [ownerUserId]`

## 当前关键运维知识

### Agent 人格/工具文件变更生效三步法

修改 `agents/<agentId>/AGENTS.md`、`TOOLS.md`、`BOOTSTRAP.md` 后，必须依次执行：

```bash
# 1. 同步到 workspace
node scripts/sync-agent-workspace.js <agentId>

# 2. 重置 systemSent
python3 -c "
import json; p='$HOME/Documents/OpenClawData/agents/<agentId>/sessions/sessions.json'
with open(p) as f: d=json.load(f)
for s in d.get('sessions',{}).values(): s['systemSent']=False
with open(p,'w') as f: json.dump(d,f,indent=2)
"

# 3. 重启 Gateway
launchctl bootout gui/$(id -u)/ai.openclaw.gateway
launchctl bootstrap gui/$(id -u) ~/Library/LaunchAgents/ai.openclaw.gateway.plist
```

缺任一步，Agent 仍会使用旧版人格/工具文件。

### 判断文件系统时间戳 vs Gateway 日志时间戳

Gateway 日志时间戳可能滞后于文件系统。以文件系统时间戳为准。

### sessions_send vs sessions_spawn 超时差异

- `sessions_send`：无硬上限，`timeoutSeconds` 直传 `agent.wait`
- `sessions_spawn`：有 120s 硬上限（`SUBAGENT_ANNOUNCE_TIMEOUT_MS = 12e4`）

## 未尽任务

1. **多条消息一股脑返回** 仍未解决
   - 当前虽已打通 `laojun -> star-steps -> laojun`
   - 但 agent-to-agent 自动播报仍会产生：
     - `NO_REPLY`
     - `REPLY_SKIP`
     - `ANNOUNCE_SKIP`
   - 群里仍可能出现多段状态集中返回，而不是自然的逐步 checkpoint
   - 这是当前最明确的未收口项

2. **项目维护 Agent 自动播报噪音** 仍需框架层收口
   - 现在 `sessions_send` 后，除了业务回复，还会有自动播报回流
   - 需要后续区分：
     - 给上游 agent 的回包
     - 面向人类时间线的消息

3. **项目 Agent session store 与项目 runtime 路径分裂**
   - 项目资产在 `projects/<projectId>/`
   - sessions 在 `agents/<projectId>/sessions/`
   - 这虽然已识别，但还未统一

4. **老君对项目内问题的总结仍偏长**
   - 主链路已对
   - 但当前对人总结还比较重，后续可继续压缩为更高密度输出

7. **飞书群聊文本消息实时投递** 仍为框架限制
   - `queuedFinal=true` 导致文本批量 flush
   - `message` 工具可绕过但 LLM 不稳定遵守
   - 根本解决需要源码层面支持或 `onRunStart` 钩子

8. **老君即时确认回复** 不稳定
   - 已在 `AGENTS.md` 和 `TOOLS.md` 中加入"即时响应规则"
   - LLM 遵守率不高，纯 prompt engineering 不够
   - 可考虑 hook 层面的自动确认消息

## 当前建议

如果下一轮继续推进，优先顺序应是：

1. 先收口**多条消息一股脑返回**
   - 排查 `sessions_send` / 自动播报 / provenance 回流
   - 目标：项目维护 Agent 做到哪回到哪，而不是最后集中刷屏
2. 继续验证 `wukong`、`guanyin`、`taibai` 首轮口径与技能白名单效果
3. 把 ACP 升级后检查、补丁重放、冒烟验证纳入固定运维流程
4. 继续把高危运行态命令拦截前移到更底层权限层
5. 定义最小 Agent 协作协议与项目 Agent 回包协议
6. 探索 `onRunStart` 钩子或 Gateway 层自动确认，解决老君即时回复问题
