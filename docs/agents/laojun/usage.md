# 太上老君使用说明

## 定位

太上老君是多项目的**首席产品官入口**。

他不直接长期维护单个项目的全部工件，而是负责：

1. 在飞书项目群里接住需求入口
2. 和人类沟通产品方向、范围、优先级与验收
3. 在项目初始化时完成群绑定、免 `@` 沟通配置
4. 创建项目专属协调员，接住项目上下文与文档维护
5. 编排 `architect`、`fullstack-engineer`、`reviewer` 与 `Codex`

## 适用场景

适合以下情况：

1. 你想把一个飞书群作为长期项目群来管理
2. 你希望先由老君完成产品收敛，再把工作拆给后续 Agent
3. 你希望群里后续可以不依赖持续 `@` 机器人来沟通

不适合以下情况：

1. 只是一次性闲聊群
2. 不需要项目上下文、工件和持续推进
3. 不准备保留 `laojun` 这个入口 Agent

## 标准使用流程

### 1. 把老君拉进项目群

前提：

1. `laojun` 已在本地实例中启用
2. 对应飞书账号已完成绑定
3. gateway 已正常工作

### 2. 在群里发第一条消息

建议直接说清：

1. 这是新项目群
2. 想做什么项目
3. 当前最想推进的第一件事

老君首轮会先判断：

1. 当前群是否已绑定 `projectId`
2. 当前任务是新建类还是变更类
3. 是否需要先补 workflow 选择

### 3. 回到 Codex 侧完成接管

当老君在群里接住第一条消息后，回到当前仓库上下文，让你的 Coding Agent 继续完成：

1. 群 `groupId -> projectId` 绑定
2. 群级免 `@` 沟通配置
3. 项目专属协调员创建
4. 项目文档与工件目录初始化

你不需要手写长步骤；直接让本地 Coding Agent 基于当前仓库完成即可。

### 4. 开始项目管理

完成初始化后，推荐协作方式是：

1. 人类继续在飞书群里和老君聊产品方向、范围、优先级、验收
2. 老君负责保持产品口径清晰，并把任务整理成可执行任务包
3. 项目协调员负责维护项目上下文、工件和推进状态
4. 三剑客按职责分别处理架构、实现、审查

## 成功标志

满足以下几点，说明老君已经进入可用状态：

1. 老君能在项目群里正常接话
2. 当前群已绑定唯一 `projectId`
3. 群里后续沟通不再依赖持续 `@`
4. 项目专属协调员已创建
5. 项目文档和工件目录已落地
6. 老君能够持续汇报当前状态、下一步、风险与待拍板项

## 常见异常

### 群里能说话，但项目没推进

常见原因：

1. 当前群还没绑定 `projectId`
2. workflow 还没完成二选一
3. 项目协调员还没创建

### 群里还得持续 `@`

常见原因：

1. 群级 `requireMention` 没关
2. 群级配置没写到正确账号下
3. 还没完成当前群的放行与绑定

### 老君在群里只能泛泛而谈

常见原因：

1. 只完成了聊天接入，没完成项目初始化
2. 项目文档目录还没落地
3. 老君还没把任务交给项目协调员和专家 Agent
